# CivilHub 開発ルール

## AIエージェント向け入口

AIエージェントがこのリポジトリで作業する場合は、まず `AGENTS.md` を確認すること。

`AGENTS.md` は作業時の短い実行ルール、`CODEX.md` はCivilHubの詳細な開発ルールとして扱う。
`CLAUDE.md` が追加された場合は補助・互換用として扱い、削除せず内容の整合を確認する。

内容が矛盾する場合は、最新のユーザー明示指示、`AGENTS.md` の実行安全ルール、`CODEX.md` の詳細ルール、既存コード挙動の順に優先する。

## 1. プロジェクト概要

CivilHub は、建設業・土木施工管理向けの施工体制台帳作成支援アプリである。

主な目的:

- 施工体制ツリーの作成
- 会社情報の入力・編集
- 施工体制台帳Excelへの反映
- 再下請負通知書Excelへの反映
- 施工体系図の出力
- 帳票・添付資料の管理
- 現場ごとのデータ管理

対象ユーザーは、実際の施工管理担当者である。

そのため、内部構造のきれいさよりも、実務で迷わず使えることを優先する。

## 2. 開発の基本方針

以下を必ず守る。

- 既存機能を壊さない
- 大規模改修を一度に行わない
- 小さく変更し、小さく確認する
- 帳票Excelの見た目・セル位置を最優先する
- UIは実務者が迷わないことを優先する
- 開発者向け機能や内部処理をユーザー画面に出しすぎない
- 変更前に関連ファイルを確認する
- 推測でファイル構造やセル位置を変更しない
- 不明点は勝手に決めず、既存仕様・README・docs・サンプルファイルを確認する

## 3. Excel帳票に関する最重要ルール

施工体制台帳・再下請負通知書・施工体系図などのExcel帳票は、見た目とセル対応が最重要である。

以下を守る。

- Excel列や行を安易に増やさない
- 既存テンプレートのセル位置を勝手に変更しない
- セル結合、罫線、印刷範囲、余白、改ページを勝手に崩さない
- 帳票出力処理を変更した場合は、必ず出力Excelを確認する
- データ構造上の都合で帳票側の列を増やさない
- 内部管理項目はDBや内部データで持ち、Excelには必要な項目だけ出力する

## 4. サンプルExcel・マスターExcelの扱い

日本道路の記入済み施工体制台帳は、以下の用途に限定する。

- 基本例
- 完成品サンプル
- セル対応確認
- 取込テスト
- 様式理解

ただし、以下のようには扱わない。

- 共通マスターとして使わない
- どの現場でも使える固定データとして扱わない
- 会社マスターの初期値として勝手に流用しない
- 現場固有・会社固有の記入済みデータを汎用マスター化しない

マスター候補ファイルは、あくまで候補として扱う。
実際にマスターへ採用するかは、項目ごとに精査すること。

## 5. データ設計方針

現場ごとのデータと、共通マスターを混同しない。

現場ごとに持つべきもの:

- 工事名
- 発注者
- 元請情報
- 施工体制ツリー
- 協力会社の現場ごとの役割
- 下請次数
- 現場固有の契約・施工範囲
- 現場に紐づく帳票
- 添付資料
- 取込済みファイル

共通マスター候補:

- 会社名
- 所在地
- 代表者名
- 建設業許可情報
- 業種
- 電話番号
- 基本的な会社情報

ただし、記入済みサンプルから自動的に共通マスター化しないこと。

## 6. 施工体制ツリーの方針

施工体制ツリーは CivilHub の中心機能である。

以下を重視する。

- 追加した会社情報を後から編集できること
- ツリー上の会社情報を帳票へ反映できること
- 施工体制台帳Excel、再下請負通知書Excelと連動できること
- 親子関係、下請次数、会社の役割を分かりやすく扱うこと
- ユーザーが「どの会社がどの帳票に反映されるか」を理解しやすいUIにすること

## 7. 帳票・添付資料管理の方針

帳票取込後の契約関係帳票・添付帳票について、以下を意識する。

- 取込済み帳票を後から確認できること
- 不要になった帳票を削除できること
- 可能なら編集・差し替えできること
- 契約関係帳票タブや機能が分かりにくい場合は、無理に複雑化せず整理すること
- 実務者が「何を登録すればいいか」分かる名称・導線にすること

## 8. UI方針

UIは実務者向けにする。

- 難しい専門用語や内部用語を前面に出さない
- ボタン名は具体的にする
- 「登録」「編集」「削除」「Excel出力」「取込」など、操作結果が分かる文言を使う
- 重要機能は見つけやすくする
- 開発者向け確認ボタンやデバッグ表示は目立たせすぎない
- 不要な確認ダイアログを増やしすぎない
- ただし削除操作には確認を入れる

## 9. ファイル・フォルダ運用

受け渡し用フォルダがある場合は、以下のように扱う。

- Codexに渡す資料は、受け渡し用フォルダ内に整理する
- ファイル名は用途が分かる名前にする
- 古い版と最新版が混在する場合は、最新版を明示する
- サンプル、セル対応表、マスター候補、完成品例を混同しない
- Codexが勝手にファイル名を大きく変更しない
- 変更が必要な場合は、READMEや指示書も合わせて更新する

命名例:

- `sample_completed_施工体制台帳_日本道路.xlsx`
- `cell_mapping_施工体制台帳_v2.xlsx`
- `master_candidate_company_v2.xlsx`
- `README.md`

ただし、既にプロジェクト内で決まった命名がある場合は、そちらを優先する。

## 10. テスト・確認方針

変更後は、可能な範囲で以下を確認する。

- アプリが起動するか
- 既存画面が開くか
- 施工体制ツリーの追加・編集ができるか
- Excel取込が壊れていないか
- Excel出力が壊れていないか
- DBマイグレーションが必要な場合、既存データが壊れないか
- 例外発生時にユーザーへ分かりやすいメッセージが出るか

現時点で確認済みの基本コマンド:

```powershell
python app.py
python -c "import ast, pathlib; ast.parse(pathlib.Path('app.py').read_text(encoding='utf-8')); print('syntax ok')"
python -B -c "import app; print(app.APP_TITLE)"
```

GUI常駐を避けてUI生成だけ確認する場合:

```powershell
python -B -c "from tkinter import Tk; import app; root=Tk(); app.CivilHubApp(root); root.update_idletasks(); root.destroy(); print('app ui ok')"
```

## 11. Git運用

作業前後で差分を確認する。

- 作業前に `git status` を確認する
- 変更後に `git diff` を確認する
- 生成物・一時ファイル・キャッシュを不要にコミットしない
- 大量の無関係な整形をしない
- 関係ないファイルを触らない
- 既存のユーザー変更を上書きしない

コミットする場合は、変更内容が分かるメッセージにする。

```powershell
git status
git diff
```

## 12. Codexモデル選択方針

今後、Codexで使用可能なモデル候補は以下として扱う。

- 5.4
- 5.4 mini
- 5.5

現状、5.3は使えない前提にする。

5.5を使うべき作業:

- 既存コードの重めの改修
- DB設計変更
- SQLiteマイグレーション
- Excel帳票出力の修正
- UI構成の整理
- 取込処理の修正
- 施工体制ツリーと帳票連動
- 複数ファイルにまたがる変更
- 不具合調査

5.4 / 5.4 miniでよい作業:

- 文言修正
- README修正
- 軽微なUIラベル変更
- コメント追加
- 小さなドキュメント整理
- 単純なファイル名整理

## 13. ChatGPTとの分担方針

このプロジェクトでは、ChatGPTとCodexを併用する。

ChatGPT側で行うこと:

- 方針整理
- 仕様整理
- 指示書作成
- 改修単位の分割
- 実務目線でのレビュー
- Codexに渡す前の作業内容整理

Codex側で行うこと:

- リポジトリ調査
- 実コード修正
- テスト
- diff確認
- ドキュメント反映
- Git操作

Codexは、曖昧な大規模改修を一気に進めず、必要に応じて作業単位を小さく分けること。

## 14. context-mode向けに残したい判断

context-modeが拾いやすいよう、重要な判断はドキュメントに残す。

特に以下の判断は重要。

- 日本道路の記入済み施工体制台帳は基本例・完成品サンプルであり、共通マスターではない
- マスター候補ファイルは項目ごとに精査する
- 施工体制ツリーが中心機能
- 会社情報は後から編集できるべき
- 帳票Excelの見た目とセル位置を最優先する
- ユーザーは施工管理の実務者であり、UIは分かりやすさ優先
- 重い改修は5.5を優先する
- 既存機能を壊さず、小分けに改修する

## 15. 禁止事項

以下は禁止。

- 記入済みサンプルを無断で共通マスター化する
- Excel帳票のセル位置を推測で変更する
- 既存機能を確認せずに大規模リファクタする
- 関係ないファイルをまとめて整形する
- DB構造を変更して既存データを壊す
- テスト未確認なのに「問題なし」と断言する
- 実在しないテストコマンドを書く
- ユーザーの意図と違う方向で機能を複雑化する
- 受け渡し用ファイルを勝手に削除する
- 古いサンプルと最新版を混同する
